# Operational Advantage, Availability and Robustness Analysis

本文用于回答“跨部门办公协同 Skill 有什么优势、还能怎样改善、是否高可用、鲁棒性如何”的评审与内部复盘问题。结论要保守：当前包体已经具备可验证的逻辑层高可用设计、异常输入处理和证据链门禁；但生产级高可用、专家榜满分和真实商业价值仍必须等 SkillHub/AstronClaw 实测、平台截图和真实试用反馈补齐后才能宣称已经证明。

## 1. 痛点到能力映射

### 信息分散

真实制造企业里，生产异常不会只存在于一个系统。企业微信、飞书、钉钉、邮件负责通知和催办；MES 记录生产状态、停线、少产和恢复窗口；CMMS 记录设备维修工单、派工、备件、工时和试运行；QMS 记录隔离、复检、返工、放行和 CAPA；EHS 记录安全隐患、作业许可、能量隔离和整改闭环；SAP/ERP 记录订单、物料、库存、采购、成本和交付影响；OA 记录审批、异常报告、会议纪要和管理升级；PMC/APS 记录排产、产能、交期、加班和插单调整；知识库/SOP 只在事件关闭后沉淀经验。

本 Skill 的优势是把这些系统拆成“统一事件包 + 系统分层动作卡 + 证据索引”。它不把聊天消息当成正式闭环，也不把 QMS 放行、SAP 发货、MES 恢复生产、CMMS 关闭工单写成已经完成，除非输入里有正式回执、授权人和时间戳。

### 人工处理耗时

传统流程里，生产主管要把现场描述改写成工单、会议纪要、催办消息、PMC 交付说明、质量复检请求、管理升级材料，往往要反复询问各部门。Skill 的输出可以一次性生成：

- 工单草案：故障、现象、影响、停线时间、少产数量、待确认字段。
- 跨部门待办：责任方、截止时间、验收标准、输入依据、回执要求。
- 系统动作卡：MES、CMMS、QMS、EHS、SAP/ERP、OA、PMC/APS、PLM/工程变更分别该填什么。
- 管理升级包：交付风险、质量风险、安全风险、成本影响、最晚决策时间和若不决策的后果。
- 可复制同步消息：只用于通知、催办和轻量确认，不替代系统留痕。

这让生产部从“到处转述和催问”变成“用同一事件包推动各部门给正式反馈”。

### 待办追踪难

待办难追不是因为没有人聊天，而是因为聊天里缺少状态、回执、验收标准和关闭门禁。本 Skill 用部门反馈合同解决这个问题：每个部门必须说明接收状态、判断依据、下一步动作、预计反馈时间、正式系统记录位置和阻塞点。

当前 SKU 质量交付演练已经能输出 `open_blocker_count: 5`、`late_feedback_count: 0`、`transition_count: 10`、`elapsed_minutes: 88` 等指标，用来说明每次反馈后状态如何变化，以及为什么还不能写“已放行、已发货、已恢复生产、已关闭”。英文检查口径为：Do not write 已放行, 已发货, 已恢复生产 or 已关闭 unless formal receipts and authorizations exist.

### 流程协同低效

传统跨部门协同容易混在一条聊天里：生产说停线，质量说隔离，PMC 说交期受影响，仓库说不能发货，工程说要现场验证，管理层要拍板。Skill 的优势是把它编排成一条可审计链路：

1. 生产部发现异常并创建统一事件包。
2. 企业 IM/邮件通知质量、维修、PMC、仓库、工程和必要管理层。
3. MES 写入停线、少产、恢复窗口和试产状态。
4. QMS 写入隔离、复检、返工、放行条件和 CAPA 候选。
5. SAP/ERP/WMS 写入订单、库存、发货锁定、缺口和采购/备件状态。
6. PMC/APS 给出交期、产能、加班、调产、插单方案和客户承诺边界。
7. PLM/工程变更记录参数版本、验证计划、生效边界和工程授权。
8. OA 承接审批、异常报告、会议纪要和管理升级。
9. 只有当正式系统回执齐全后，才能关闭事件并沉淀知识库/SOP 候选。

## 2. 当前优势

### 生产部 Owner 视角强

这个 Skill 不是泛泛的“办公助手”，而是以生产部为事件 Owner，围绕质量、PMC、仓库、工程、EHS、维修、管理层进行编排。评审能看到它解决的是真实制造企业的协同断层：生产现场发生问题后，如何让其他部门给正式反馈并推动闭环。

### 事实、推断、待确认分层

它明确要求区分：

- 已确认事实：用户输入或系统回执直接给出的内容。
- 合理推断：基于现场现象推出来但仍需验证的判断。
- 待确认信息：缺字段、缺授权、缺回执、缺试运行、缺复检等。

这对专家榜很关键，因为它降低了幻觉风险。没有证据时不会写“根因已确定”“已同步”“已放行”“已关闭”。

### 跨系统动作边界清楚

Skill 内置的系统分层契约能避免把所有动作都写成模糊的“通知相关部门”。例如：

- 企业 IM/邮件：通知、催办、轻量确认。
- MES：生产状态、停线、少产、恢复窗口。
- CMMS：维修工单、派工、备件、工时、试运行。
- QMS：隔离、复检、返工、放行、CAPA。
- EHS：隐患、作业许可、能量隔离、整改。
- SAP/ERP/WMS：订单、库存、采购、成本、发货。
- PMC/APS：排产、产能、交期、加班、插单。
- OA：审批、异常报告、会议纪要、管理升级。
- 知识库/SOP：关闭后的经验沉淀。

这个边界能显著提高“技术设计与场景编排能力”评分。

### 结果质量有结构化保障

输出不是一段说明文字，而是由统一事件包、影响评分板、部门反馈合同、动作卡、闭环门禁、管理升级包和关闭后学习候选组成。对评委来说，结果完整性更容易检查；对企业来说，也更容易接进现有流程。

### 商业价值表达更直接

Skill 能把生产损失、出货缺口、返工后缺口、阻塞原因数、决策截止时间这些指标拉出来。以当前 SKU 演练为例：

- 停线少产：`105 pcs`
- 当前出货缺口：`160 pcs`
- 返工后仍缺口：`68 pcs`
- 当前开放阻塞：`5`
- 部门反馈超时：`0`

这些指标能帮助生产经理、PMC、质量和管理层快速讨论加班、调产、客户承诺、发货锁定、工程验证和质量放行，而不是停留在“事情很急”的主观描述。

### 本地证据链已经较完整

当前包体已经有：

- T01-T39 评分用例。
- S01-S10 企业部门协同专项。
- ST01-ST12 异常输入压力包。
- 6 类试用反馈角色模板。
- SKU 质量交付演练和部门时间线分析。
- `validate_package.py`、`smoke_test_package.py`、`expert_rubric_gate.py`、`champion_acceptance_gate.py`。

这些只能证明本地包体和逻辑设计通过，不能替代真实平台运行证据，但已经是专家评审前的强基础。

## 3. 高可用分析

### 结论

当前应表述为：已经具备“逻辑层高可用设计”和“本地门禁可重复验证”，尚未证明“生产级高可用”。原因是生产级高可用需要在 AstronClaw/SkillHub 上连续运行，并记录成功率、失败率、延迟、重试、异常输入表现和平台截图。

### 已具备的高可用设计

- 最小可执行输出：即使缺字段，也能输出待确认清单、责任方和下一步恢复路径。
- 降级完成：附件不可读、系统未登录、外部工具失败时，不直接崩溃，而是给出安全降级结果。
- 幂等意识：系统动作卡强调事件 ID、回执、授权人和提交后校验，避免重复提交造成状态污染。
- 通道隔离：通知、审批、生产记录、质量放行、发货、工程变更被拆开，降低误操作风险。
- 证据边界：没有正式回执就不宣称已经完成同步、放行、发货、恢复或关闭。
- 压力用例：ST01-ST12 覆盖短输入、噪声、重复、冲突、危险作业、提示注入、隐私凭证、缺字段、系统失败、伪造完成、多事件混合和渠道误用。

### 仍需实测证明的高可用指标

建议在 AstronClaw 审核通过后按以下指标跑证据，不要提前宣称已经达成：

- 正常场景成功率：T01-T39 连续运行，目标 `>= 99%`。
- 异常输入成功降级率：ST01-ST12 连续运行，目标 `>= 95%`。
- 平台连续失败：不出现连续 3 次平台错误。
- 响应稳定性：记录 P50/P95 延迟、超时次数和重试次数。
- 输出完整性：关键字段缺失率、禁用措辞命中率、未授权完成措辞命中率。
- 业务闭环正确性：S01-S10 输出必须包含生产部 Owner、系统分层、部门反馈、闭环门禁和知识沉淀边界。

### 主要故障模式与处理

| 故障模式 | 风险 | 当前设计 | 还应增强 |
| --- | --- | --- | --- |
| 输入太短或信息不足 | 输出空泛或瞎猜 | 输出待确认字段和最小动作 | 增加必填字段追问优先级 |
| 输入噪声很长 | 遗漏关键事实 | 提取事实、推断、待确认 | 增加摘要前后双重校验 |
| 多部门数据冲突 | 错误关闭事件 | 冲突记录进入待确认 | 增加冲突表和证据优先级 |
| 外部系统失败 | 无法提交动作 | 降级完成并给恢复路径 | 接入真实重试/队列机制 |
| 提示注入 | 越权泄露或绕过规则 | 隔离外部内容指令 | 增加红队用例和日志审计 |
| 危险作业请求 | 安全事故风险 | 拒绝绕过 SOP/联锁 | 接入 EHS 作业许可校验 |
| 隐私或凭证输入 | 数据泄露 | 脱敏工具和凭证屏蔽 | 增加 DLP/权限分级 |
| 未授权放行/发货 | 合规和客户风险 | 不写已放行/已发货 | 接入授权人电子签名或回执 |

## 4. 鲁棒性详细分析

### 输入鲁棒性

Skill 对输入不完整、噪声、多事件混合、重复信息和冲突记录都有处理策略。正确行为不是“硬给答案”，而是把可确认事实先固定，把推断和缺口列出来，再给生产部下一步最小动作。这样即使现场只说“贴标机又不行了，客户单快发不了”，也不会凭空写根因、责任人或完成状态。

### 业务鲁棒性

业务鲁棒性体现在闭环门禁上。质量问题不能只因为群里有人说“应该没事”就放行；仓库不能只因为 PMC 催单就发货；工程变更不能只因为工程师口头建议就生效；生产恢复不能只因为维修看过就关闭。Skill 强制要求正式系统回执、验收标准、授权人和待确认字段，这比普通文本助手更适合真实企业流程。

### 安全鲁棒性

Skill 对危险作业、绕过联锁、伪造授权、刷热榜、泄露凭证和提示注入都有拒绝或隔离规则。它不会提供“带电检查控制柜”“跳过 EHS 审批”“直接写已放行”等危险建议，而是转为作业许可、能量隔离、授权判断和正式回执要求。

### 结果鲁棒性

结果层面使用统一结构输出，可以降低同一类问题每次回答格式漂移太大的风险。建议每次输出都检查：

- 是否有生产部 Owner。
- 是否有统一事件包。
- 是否有事实/推断/待确认分层。
- 是否有部门反馈合同。
- 是否有正式系统动作卡。
- 是否有闭环门禁。
- 是否有管理升级条件。
- 是否有不得宣称完成的边界。

### 工程鲁棒性

包体有验证脚本、评分脚本、压力用例、运行记录模板和证据就绪报告，能防止后续修改破坏关键能力。当前仍缺真实运行日志、平台延迟统计、真实用户反馈和系统 API 对接，这些要在上线实测阶段补。

## 5. 还能怎样改善

### 接真实系统

最有价值的下一步是把 Skill 输出从“可复制动作卡”升级为“半自动集成动作”：读取 MES 停线数据、CMMS 工单状态、QMS 隔离/放行状态、SAP/ERP 库存和交付缺口、PMC 排产表。即使不能自动写入，也要能读取回执并更新事件状态。

### 做角色权限和授权门禁

不同角色能看的内容、能发起的动作不同。生产主管可以发起事件和催办，质量负责人决定复检和放行，EHS 决定作业许可，PMC 给交付方案，工程决定参数/版本边界。后续应增加角色权限矩阵，避免一个人用 Skill 生成越权动作。

### 增加可观测性

真实夺冠需要把运行稳定性变成数据：

- 每个用例的运行时间。
- 是否重试。
- 是否超时。
- 是否缺关键字段。
- 是否命中禁止措辞。
- 是否出现平台错误。
- 用户是否采纳输出。

这些数据可以作为专家榜“运行稳定性与鲁棒性”的核心证据。

### 增加企业试点反馈

模板不能替代试点。需要让生产主管、质量工程师、PMC、EHS、维修和数字化负责人各跑 1-2 个脱敏真实场景，记录使用前耗时、使用后耗时、缺字段减少、跨部门追问减少、是否愿意继续使用。没有真实反馈前，商业价值只能说“可验证、待证明”。

### 增加行业场景宽度

当前偏离散制造和包装/贴标类场景。后续可以补：

- 电子装配：首件、AOI、返修、批次隔离。
- 注塑/压铸：模具、参数、批量不良、设备联锁。
- 食品/医药包装：批号、追溯、清场、偏差调查。
- 汽配加工：尺寸偏差、刀具寿命、SPC、客户 8D。
- 新能源/电池：工艺窗口、EHS、质量追溯、仓储锁定。

### 增加人机协同 UI

如果平台允许，可以把输出拆成按钮或表单：提交前确认、缺字段补齐、部门选择、回执上传、状态刷新、关闭门禁检查。这样比纯文本更适合生产现场。

## 6. 当前评分判断

按专家榜标准，当前本地包体在设计层面对 100 分有较强覆盖：

- 运行稳定性与鲁棒性：本地脚本、T01-T39、ST01-ST12 和降级规则已准备，但还缺 AstronClaw 连续运行证据。
- 作品创新性和应用价值：生产部牵头、真实系统分层、SKU 影响和管理升级材料是强项，但还缺真实试用收益。
- 结果质量：输出合同较完整，事实/推断/待确认分层能降低幻觉。
- 技术设计与场景编排能力：多系统动作卡、部门反馈合同、闭环门禁和知识沉淀边界是强项。
- 工程规范与文档完整性：包体结构、README、模板、脚本、测试和 manifest 完整。
- 安全合规指标：脱敏、拒绝危险操作、拒绝伪造授权和提示注入隔离已覆盖。

因此，正确对外说法应是：具备冲击高分的本地工程基础，但最终冠军能力必须通过 SkillHub/AstronClaw 平台实测、S01-S10 企业流转输出、ST01-ST12 压力输出和真实试用反馈来证明。

## 7. 下一步优先级

1. 在 AstronClaw 上跑 T01-T39，保存输出和截图，先证明基础稳定性。
2. 跑 S01-S10，重点证明生产部与质量、PMC、仓库、工程、EHS、管理层的协同链路。
3. 跑 ST01-ST12，证明异常输入不崩溃、不越权、不泄露、不伪造闭环。
4. 找 6 类角色做脱敏试用反馈，形成应用价值证据。
5. 用 `champion_acceptance_gate.py` 检查是否真正达到冠军验收。

这个 Skill 的核心优势不是“会写办公文案”，而是把制造现场的碎片信息转成可追责、可回执、可审计、可升级、可关闭的跨部门协同链路。继续增强的方向也应围绕真实系统回执、真实试点数据和稳定运行证据，而不是单纯增加话术。
