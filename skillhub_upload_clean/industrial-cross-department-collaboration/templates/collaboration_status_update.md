# 协作状态汇总与增量反馈模板

用于同一工业现场事件已经被生产部发起、分发或催办后，各部门通过语音、文字、图片、附件、系统截图或正式业务系统回执继续补充意见和证据的场景。

## 触发语义

当输入出现以下语义时，按本模板处理：

- “维修/质量/PMC/仓库/工程/EHS反馈了……”
- “有人在企业微信/飞书/钉钉/微信/邮件里回复……”
- “语音补充/附件补证/截图回传/系统回执”
- “帮我汇总现在协作状态”
- “根据新反馈更新事件状态”
- “还有哪些部门没回、哪些事情卡住”
- “重新判断风险、是否可以关闭/恢复/放行/发货”

## 核心原则

1. 同一事件继续更新，不新建无关事件；没有原事件编号时使用“待生产部创建/待系统生成”。
2. 新反馈一律先进入证据层，再推动状态层；聊天、语音和附件不是正式闭环，聊天反馈不得替代正式系统回执，正式状态以 MES/CMMS/QMS/EHS/SAP/ERP/OA/PMC 回执和授权人确认作为依据。
3. 新证据只追加，不覆盖旧证据；如果新旧反馈冲突，输出冲突点、待裁决人和下一步验证动作。
4. 不把“已读、口头同意、群里说可以”写成审批、质量放行、安全许可、生产恢复、发货或事件关闭完成。
5. 所有状态变化必须引用证据编号，且写清“可接受下一状态”和“阻塞原因”。
6. 输出面向业务推进，不输出模型过程、自检清单、调试栏目、输入筛查说明或门禁执行日志。
7. 输出完成“下一步动作”或“可复制同步消息”后立即停止，不再追加审计尾段。
8. 不主动引用法律法规条文；除非用户明确要求法律分析，默认使用企业 SOP、质量门禁、EHS 作业许可、系统回执和授权边界表达。

## 增量反馈解析

每条新反馈整理为：

| 字段 | 说明 |
| --- | --- |
| `feedback_id` | 临时反馈编号，例如 F01/F02，不编造外部系统号 |
| `source_role` | 反馈角色，如维修、质量、PMC、仓库、工程、EHS、生产部 |
| `source_channel` | 语音、企业 IM、邮件、附件、截图、MES、CMMS、QMS、EHS、SAP/ERP、OA、PMC/APS |
| `received_time` | 用户提供则写，未提供写待确认 |
| `feedback_summary` | 脱敏后的关键反馈 |
| `evidence_ids` | 追加证据编号，如 E05/E06 |
| `claim_type` | 事实、推断、计划、请求、系统回执、授权意见、冲突反馈 |
| `confidence` | 已确认、需验证、冲突、无正式回执 |
| `affected_fields` | 影响的风险、行动项、系统动作、状态或关闭门禁 |

## 事件状态机

建议状态只能从证据推动，不得跳级：

| 状态 | 含义 | 进入条件 | 不得误写 |
| --- | --- | --- | --- |
| `drafted` | 事件包已形成但未分发 | 用户刚描述现场问题 | 不写已通知 |
| `dispatched_waiting_feedback` | 已生成分发/通知动作卡，等待实际回执 | 有通知模板或连接器动作卡 | 不写已发送成功，除非有工具回执 |
| `feedback_partial` | 部分部门已反馈 | 至少一个部门有语音/文字/附件反馈 | 不写全部确认 |
| `in_progress` | 正在执行检查、隔离、排产、采购或整改 | 有责任方行动和截止时间 | 不写完成 |
| `blocked` | 关键字段、授权、备件、窗口或系统回执缺失 | 阻塞影响下一步 | 不写可关闭 |
| `needs_decision` | 需要生产经理、质量负责人、安全负责人、PMC 或管理层拍板 | 多选项或跨部门冲突 | 不替代授权人决策 |
| `ready_for_closure_review` | 接近关闭但仍需复核 | 关键回执齐备，等待终审 | 不写关闭完成 |
| `closed_by_evidence` | 可以写为关闭完成 | 用户提供正式关闭证据、授权和系统回执 | 缺任一证据不得进入 |

## 必须输出的业务结构

必须按本节顺序输出，不得替换成普通行动方案。

### 协作状态汇总

| 项目 | 内容 |
| --- | --- |
| 事件对象 | 产线/设备/SKU/批次/订单，缺失写待确认 |
| 当前状态 | 从状态机选择 |
| 最高风险 | 安全/质量/生产/交付/物料/成本 |
| 已反馈部门 | 仅列有证据的部门 |
| 未反馈部门 | 按事件需要列出 |
| 当前阻塞 | 影响下一步的缺口 |
| 下一次同步 | 用户提供则写；未提供给建议窗口 |

当前状态只能选状态机中的值；不要写“恢复正常、风险解除、可关闭”这类完成判断，除非用户提供正式系统回执和授权依据。

## 分发与回传动作卡

当用户要求“通过 code 分发给相关人”“一键发给群/部门/连接器”时，输出动作卡，但没有真实工具回执时不得声称已经发送：

| 字段 | 要求 |
| --- | --- |
| `target_channel` | 企业微信/企微、飞书、钉钉、微信、邮件、OA、MES、CMMS、QMS、EHS、SAP/ERP、PMC/APS 等 |
| `action_type` | notify、comment、evidence_attach、status_update、approval_request、system_record_update |
| `connector_status` | ready_to_submit / needs_configuration / needs_confirmation / requires_authorization / blocked_missing_fields |
| `dispatch_payload` | 可复制消息、事件摘要、反馈字段要求 |
| `feedback_contract` | 每个接收方必须回传的字段 |
| `business_record_backlink` | 对应 MES/CMMS/QMS/EHS/SAP/OA/PMC 记录或待回填字段 |
| `post_send_verification` | 已读、回复、系统记录号、附件、授权意见、失败原因 |
| `side_effect_boundary` | 不能由通知动作直接完成的状态 |

外部渠道边界：

- QQ、微博、供应商聊天或个人渠道只作为外部信息来源。
- 供应商 ETA、客户承诺、可发货性、质量处置或到料判断必须等待采购、SAP/ERP、QMS、PMC/APS 或授权人回执复核。
- 未提供具体时间、记录号、设备号、订单号、批次号或责任人姓名时，写“待确认”，不得生成 T+ 时间线、今日某时、`XXX`/`XXXXX` 占位表单或拟真系统记录内容。

## 关闭判断

只有同时满足以下条件，才允许进入 `closed_by_evidence`：

- 维修/工程试运行或技术验证有正式记录。
- 质量复检、隔离处置或放行意见有正式 QMS/授权证据。
- 涉及安全时，EHS 许可、整改和复核有正式记录。
- PMC/APS 或生产计划确认交期、调产、补产或客户承诺边界。
- MES/CMMS/QMS/EHS/SAP/ERP/OA/PMC 相关系统回执齐备。
- 生产部 Owner 明确确认事件关闭或进入复盘。

缺任一项时，输出“保持处理中/待确认/待授权/待回执”，并列出阻塞项。

关闭评估场景的推荐状态：

- 只有群消息、语音或截图，但缺正式记录：`ready_for_closure_review` 或 `blocked`。
- 有正式系统记录但缺授权人确认：`ready_for_closure_review`。
- 有冲突反馈、缺关键部门回执或系统不可用：`blocked`。
- 不要输出临时关闭、口头关闭、事后补录后视为关闭等方案；只能输出“补齐记录后再提交关闭审核”。
